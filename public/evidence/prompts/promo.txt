# 同题任务：为《一脚晋级》制作一张真正可玩的酷炫交互宣传页

【设计意图】

不是做一张“足球背景 + 三张功能卡”的普通落地页，而是把《一脚晋级》的核心乐趣直接做进网页：访客进入首屏就能在一个缩小版 6×6 球场里滑动足球、判断路线、三步破门，再顺着滚动叙事理解 500 关怎样逐层加入足球规则，最后自然走到“微信搜《一脚晋级》”。页面要像一款成熟微信小游戏的发布现场，清楚、友好、有比赛日荣誉感，也要有让人愿意反复拖动和探索的前端表现力。

动手前先用一句话复述你理解的任务边界，然后直接继续完成，不要等待回复。

【工作目录与必读文件】

- 只在当前目录工作。当前目录里的 `SPEC.md` 就是本任务书，必须完整遵守。
- 当前目录已经预置 `assets/`，这是三位参评者逐字节一致的唯一素材包；可以自由裁切、组合、加 CSS 遮罩，但不得修改原素材。
- 素材含义：
  - `assets/brand-lockup.png`：正式《一脚晋级》品牌 Logo。
  - `assets/stadium.jpg`：晴空绿茵比赛日竖版背景。
  - `assets/gameplay-start.png`、`gameplay-pass.png`、`gameplay-win.png`：同一真实关卡的开始、移动后、三星晋级状态。
  - `assets/qr.png`：正式小程序码；页面只作为本地原型展示，不声称已经重新实扫验证。
  - `assets/player.png`、`assets/keeper.png`：队友与门将透明角色素材。
  - `assets/route-card.png`、`assets/revive-card.png`：路线卡与保连卡道具素材。
- 工作区里出现其他文件或改动是正常的；不要读取、修改或回滚父目录、兄弟目录及其他参评者的产物。

【任务】

1. 制作一张中文单页交互宣传页，公开名称只能写《一脚晋级》，不得出现内部项目名。交付至少包括 `index.html`、`styles.css`、`app.js`，本地 HTTP server 打开即可完整体验。
2. 只能使用原生 HTML、CSS、JavaScript、Canvas 或内联 SVG，以及当前目录的本地素材；禁止安装依赖、联网、CDN、外部字体、外部图片、外部音视频和外部代码。
3. 视觉必须遵循项目的“晴空绿茵比赛日”语言：
   - 页面阅读层以奶油白 `#F3F8EF`、浅绿 `#E8F1E4`、深墨绿 `#1C3A24` 为主；球场可用 `#2E7D43`。
   - 正向主操作使用草场绿 `#2FA35A`，金色 `#F4C430` 只表达奖杯、奖励、晋级和高价值动作，红色只表达失败或危险。
   - 整体成熟、清楚、友好，有比赛日荣誉感；禁止赛博朋克、廉价霓虹、大面积纯黑、AI 紫蓝渐变、渐变标题、默认组件拼盘、emoji 和自定义鼠标光标。
   - 允许有强烈构图、质感、光影、视差和动效，但不能牺牲文字可读性。
4. 首屏必须达到“能直接玩”的标准：
   - 使用非对称桌面构图，品牌文案与可玩的 6×6 路线挑战同时在首屏主要区域可见；不是点击 CTA 后才出现。
   - 挑战容器必须有稳定选择器 `#demo-board`，可以用 DOM、Canvas 或 SVG 实现。
   - 初始足球位置为 `(row:4,col:0)`，球门为 `(row:1,col:5)`；固定障碍至少包含 `(4,5)` 与 `(0,4)`，还可加入不会破坏解法的队友、对手、泥地或装饰。
   - 每次向 `up/down/left/right` 滑动，足球沿行或列滑到障碍前、边界或途中球门；标准解法必须是 `right → up → right`，三步进球。
   - 桌面支持真实鼠标拖拽或滑动，手机支持真实触摸滑动；同时提供清楚的键盘或按钮降级路径。不能只有测试 API 能玩。
   - 拖动时要有方向/路径预览，移动时有可信的速度、缓动和落点反馈；进球后出现收束明确的晋级庆祝、步数反馈与重新挑战入口，不刷新页面。
5. 长页至少包含以下四个有明确 `id` 的叙事段落，视觉结构不要全部做成等宽卡片：
   - `#play`：首屏与三步破门挑战。
   - `#journey`：表现“500 关，不是重复堆量”。可点击或滚动探索“基础移动、泥地、转向箭头、越位、队友/对手、对手门将、黄牌、传送门”怎样逐层加入；不得编造玩家数、下载量或留存数据。
   - `#proof`：使用三张真实截图制作可操作的前/中/晋级展示，可用轮播、拖拽、叠卡、遮罩对比或其他原创交互；三张图都必须能被访客看到。
   - `#play-now`：显示正式小程序码、文字“微信搜《一脚晋级》”与一个明确主 CTA。二维码旁注明“本地原型展示，发布前需微信实扫确认”。
6. 页面必须讲清三个真实卖点，但文案与版式可自行创作：
   - 一滑到底，先看路线再动手。
   - 500 关逐层加入规则，每关程序验证有解。
   - 泥地、转向、越位、队友/对手、对手门将、黄牌与传送门会改变路线判断。
7. 动效与交互：
   - 至少有首屏入场、滚动叙事、按钮按压、挑战路径反馈、进球庆祝和截图交互六类可感知反馈。
   - 自动循环动效必须提供可访问的暂停/继续按钮，并尊重 `prefers-reduced-motion`；暂停后持续动画确实停止。
   - 只持续动画 `transform`、`opacity`、`filter` 或 `clip-path`，不要用循环改 `width/height/top/left`。
   - 不得闪烁超过每秒 3 次；不要用大面积高模糊霓虹光晕冒充“酷炫”。
8. 响应式与可访问性：
   - 1440×900、1280×720、390×844、320×568 均不得横向溢出，关键内容、挑战、二维码和主 CTA 不得被裁切。
   - 触控目标至少 44×44px；有语义化按钮、键盘焦点、合理的标题层级、图片 `alt` 与动态结果 `aria-live="polite"`。
   - 图片加载失败时仍保留可读内容与本地 CSS/几何回退，不能出现透明空洞或开发占位字样。
9. 稳定性与性能：
   - 无未处理异常、console error、重复事件绑定、失控定时器或外部网络请求。
   - 页面失焦时暂停持续动效；重置挑战不能叠加动画循环。
   - 首次打开即可阅读和操作，不得有阻挡整页的加载遮罩；可用轻量骨架或资源状态提示，但本地素材加载完要自然消失。
10. 提供统一测试接口 `window.__ONEKICK_TEST__`，至少包含：
   - `snapshot()`：返回可序列化对象，至少含 `phase`、`moves`、`ball:{row,col}`、`goal:{row,col}`、`interacted`、`motionPaused`、`activeSection`、`assetsReady`。
   - `resetDemo()`：回到初始球位和 0 步，停止尚未完成的移动/庆祝动画。
   - `move(direction)`：接受 `up/down/left/right`，走与真实输入相同的移动与胜负逻辑；可以返回 Promise，但调用结束后快照必须稳定。
   - `setMotionPaused(paused)`：真正暂停或继续所有自动循环装饰与自动轮播，不影响访客手动操作挑战。
   - `goToSection(id)`：滚动到 `play/journey/proof/play-now` 之一并更新 `activeSection`。
11. 完成后自行用本地 HTTP server 做真实浏览器检查；可以留下必要的轻量自测脚本，但不要留下浏览器 profile、录屏、依赖目录或无关文件。

【验收判据】
